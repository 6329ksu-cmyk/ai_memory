## 1. 요청 분류 (최우선)
사용자 요청을 받으면 **먼저 유형을 판단**하라:

| 유형 | 예시 | 행동 |
|------|------|------|
| **코딩/개발** | 구현, 버그, 리팩토링 | 프로젝트 루트의 `DEVLOG.md` 읽고 작업. 아래 §2 따름 |

| **애매함** | | "코드 작업인가요, 기억/맥락 확인인가요?" 먼저 물어라 |

## 2. 코딩/개발 규칙

### 프로젝트 진입
1. 프로젝트 루트에 `DEVLOG.md`가 있는지 확인
2. **있으면** → 읽고 현재 상태 파악 후 작업
3. **없으면** → `C:\dev-prompts\DEVLOG.md` 템플릿을 프로젝트 루트에 복사 → 상태 섹션 채움

### 작업 중
- 파일 수정 전 해당 파일을 먼저 읽어라
- 요청받은 것만 수정. 관련 없는 코드 건드리지 마라
- 기존 코드 스타일 유지. 새 패턴/라이브러리 임의 도입 금지
- `DEVLOG.md`의 `[잠금]` 파일은 수정 전 사용자 확인 필수

### 작업 완료 (필수)
`DEVLOG.md`에 변경 기록 추가. **기록 없이 끝내지 마라.**
```
### YYYY-MM-DD | 작업 제목
- **유형**: feature | bugfix | refactor | config | hotfix
- **파일**: 수정한 파일
- **내용**: 1줄 요약
- **영향**: 다른 파일/기능에 미치는 영향. 없으면 "없음"
```

### 버그 발생
`DEVLOG.md` → `[이력]` 최근 항목 → `[미해결]` 순서로 확인

## 4. 작업 로그 (CHANGELOG)
- 작업을 완료하면 `C:\AI_Memory\CHANGELOG.md`에 한 줄 기록을 남겨라
- 형식: `- [도구명] 작업 내용 요약`. 세부 항목이 필요하면 하위 목록으로 써도 됨
- 이 파일은 사람이 보는 용도. **AI가 매 세션마다 읽지 마라.** 사용자가 "로그 보여줘" 하면 그때만 읽어라
- 그래프(memory.jsonl)에는 핵심 맥락만, CHANGELOG에는 작업 내역을 남겨라
- **용량 관리**: 월 단위로 섹션 구분(`## YYYY-MM`). 3개월 이상 지난 섹션은 삭제해라
- **로그 분석 요청 시**: 전체를 읽되, 답변은 요약으로. 원문 복붙하지 마라. 기간별/도구별/유형별로 정리해서 보고하라

- 확신 없으면 물어라

### 서브에이전트 위임 (Opus + Sonnet)
Max 플랜 전환(2026-04) 이후 외부 AI(Gemini/Antigravity/Cursor) 위임은 폐지.

- **Opus(나)** = 설계·판단·검수·방향 수정. 단순 실행에 Opus 토큰 낭비 금지
- **Sonnet 서브에이전트** = 코드 작성·탐색·테스트·파싱 등 실무
  - `Agent` 도구 + `model: "sonnet"` 으로 위임
  - 독립 작업 여러 개면 **병렬 동시 실행** (서브에이전트마다 독립 컨텍스트 → 과부하 방지)
  - prompt 첫 줄에 "§0 응답 규칙 준수" 명시

**핵심:** Opus 토큰은 고급 자원. 단순 실행은 Sonnet으로 떼어내라.

## 7. 상황판 (PROJECT_STATE.md) 규칙
프로젝트 루트에 `PROJECT_STATE.md`가 **있으면** 이 규칙 적용. 없으면 무시.

### 7.1 작업 완료 시 트리 갱신 — "AI 선언 → 사용자 승인 → 기록"
1. 논리적 작업 단위를 끝냈다고 판단 시 **"이 작업 끝난 것 같은데 맞아요?"**로 선언
2. 사용자 `응/ㅇㅇ/ㄱ/맞아/다 됐어` 등 승인
3. 승인 **직후** `PROJECT_STATE.md`의 `▼ 트리` 섹션을 Edit 도구로 갱신
4. 노드 형식:
   ```
   ├─ 📌 {노드 이름} [v{N}] {상태}
   │   ↳ 왜: {1줄 의도}
   │   ↳ 파일: {수정한 파일들}
   ```
5. 상태: ✅완료 | ⏳진행중 | ❌미시작 | ⏸️보류
6. 노드 이름은 AI가 자동 작명. 사용자 승인 불요(이상하면 나중에 수정)

### 7.2 버전 스탬프
- 새 노드 추가 → `v1`
- 같은 노드 재작업 → `v2 → v3 → ...` 카운트 업
- 프론트매터 `last_updated` / `version` 필드도 함께 갱신
- 변경 이력 섹션에 한 줄 추가: `- v{N} (YYYY-MM-DD) — {요약}`

### 7.3 5회 수정 경고 (AI 루프 탈출용)
같은 노드가 `v5`에 도달하면 AI 스스로 경고:
> ⚠️ 이 노드 5회째 수정 중. 근본 원인 못 찾고 땜질 중일 가능성. 접근 재검토 필요.

사용자 우유부단 감지 X. AI 자기 루프 탈출용 안전장치.

### 7.4 용어 번역 의무
`PROJECT_STATE.md`에 등장하는 **모든 전문 용어**에 사용자 언어로 병기.
예: `노드(= 트리의 한 줄 항목)`, `git(= 코드 변경 이력 저장소)`, `임계값(= 경고 기준 횟수)`

### 7.5 DEVLOG.md와 차이
- `DEVLOG.md`(§2) = 코드 변경 이력 (파일/유형/영향)
- `PROJECT_STATE.md`(§7) = 의도/기능 트리 (노드/버전/상태)
- 둘 다 있으면 **둘 다 갱신**. 없는 쪽은 생략.

### 7.6 세션 시작 자동 렌더 (상황판 + 뉴런)
프로젝트 루트에 `PROJECT_STATE.md`가 있고, 해당 프로젝트의 **첫 사용자 메시지**에 응답할 때, 응답 앞부분에 아래 포맷으로 렌더링하라.

#### 7.6.1 프로젝트 식별 (AI 추측 금지)
다음 순서로 기계적으로 식별:
1. `PROJECT_STATE.md` frontmatter의 `project:` 필드 ← 최우선
2. CWD basename (폴더명) ← fallback
3. frontmatter에 `neuron_entities: [이름1, 이름2]` 있으면 그 엔티티 강제 포함

#### 7.6.2 뉴런 자동 조회
1. `mcp__memory__search_nodes(프로젝트명)` 호출 → 관련 엔티티 수집
2. `mcp__memory__search_nodes(CWD basename)` 호출 → 추가 수집
3. `neuron_entities` 목록이 있으면 `mcp__memory__open_nodes([...])` 로 강제 로드
4. `User_Plan_Status` 등 전역 엔티티는 항상 포함 (프로젝트 무관 사용자 프로필)

#### 7.6.3 렌더 포맷
```
📋 상황판 ({프로젝트명})
🎯 목표: {🎯 섹션 한 줄}

▼ 트리
{모든 노드 — [v버전] 상태. '↳ 왜' '↳ 파일' 라인은 접음}

❓ 미해결: {있으면 한 줄}

🧠 뉴런 (이 프로젝트 연결)
📍 식별 키: project="{값}" / cwd="{폴더명}"
📍 수동 연결: {neuron_entities 있으면 나열, 없으면 생략}

1. {엔티티명} ({observation 개수}개 관찰)
   - {observation 1}
   - {observation 2}
   - ... (전부 나열, 접지 말 것)

🌐 전역 뉴런 (항상 참조)
- User_Plan_Status: {짧은 요약}
- (기타 전역 엔티티)

"이어서 할까요, 새 작업 할까요?"
```

#### 7.6.4 동작 규칙
- 한 세션에 **1번만** 렌더. 이후 메시지엔 생략
- `PROJECT_STATE.md` 없으면 이 규칙 무시 (상황판만 없어도 skip, 뉴런은 사용자가 직접 요청 시만)
- 사용자가 "상황판 줘" / "뉴런 보여줘" 요청 시 언제든 재렌더
- 잘못 매칭된 뉴런은 사용자가 "이거 아냐" 하면 제거 또는 `neuron_entities` 재정의

#### 7.6.5 토큰 철학
Max 플랜 기준. 초보 사용자의 **학습·검증**이 토큰 절약보다 중요. 과감하게 전체 observation 덤프하라. 오래된 뉴런이 리포트를 지저분하게 만들면 `뉴런 정리해` 명령으로 가지치기.

## Memory Protocol (Claude Code 전용) — 뉴런(Neuron)
공용 기억 그래프 별칭: **뉴런(Neuron)**. 대화 중 '뉴런에 기록해', '뉴런 조회', '뉴런 정리해' 등으로 지칭 가능.
저장 위치: `G:\내 드라이브\AI_Memory\memory.jsonl` (홈랩) / `C:\AI_Memory\memory.jsonl` (회사)

### 조회 규칙
- **세션 시작 시 read_graph 자동 호출 금지.** 전체 그래프 로딩은 토큰 낭비.
- 이전 맥락·프로젝트 언급 시 `search_nodes("키워드")` 또는 `open_nodes(["엔티티명"])`로 필요한 것만 조회.
- Windows 스크립트(.ps1/.sh/.py) 작성 전 `search_nodes('Known_Issues_Windows')` 필수 조회.

### 자동 저장 기준 (다음 4가지만 자율 기록)
1. 사용자 선호·성향 (작업 방식, 커뮤니케이션 스타일)
2. 프로젝트 핵심 맥락 (목표, 스택, 제약)
3. 반복 이슈/해결책 (Known_Issues_* 엔티티)
4. 인프라 상태 변경 (도구 추가/설정 변경/경로 이동)

### 저장 금지
- 일반 코드 지식 (검색으로 복구 가능)
- 한번 쓰고 버릴 중간값
- 추측성·미검증 정보
- 원문 통째 복사 (핵심만 원자 단위로)

### 검토 트리거
- **세션 내 3건 이상 기록 시**: 세션 종료 직전 "오늘 뉴런에 이거 추가했는데 유지/수정/삭제?" 요약 제시.
- **엔티티 observation 15개 초과 시**: "뉴런 ○○ 비대, 정리 권장" 알림.

### "뉴런 정리해" 명령
전체 read_graph → 중복·오래된 항목 목록 제시 → 사용자 승인 후 병합/삭제 반영.
